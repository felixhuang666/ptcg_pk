## Root Cause
The internal array for map drawing and representation maintained an outdated convention of treating `2` as the grass tile while storing it as 0-indexed (subtracting `1` when reading user inputs, and adding `1` when rendering to the Phaser layer). When loading an explicit `id: 1` structure from metadata files like `main_20x10.json`, the logic mathematically forced the sprite texture frame to resolve to `id=2` (displaying the second tile instead of the intended first tile). Empty tiles and eraser variables were additionally using `-1`, requiring complex +1 offset tracking across both `ground` and `object` map layers.

## Solution
1. Rewrote indexing mathematical transforms to inherently assume a 1-based structure that flawlessly matches the server JSON payload structure and `main_20x10.json` metadata rules.
   - Removed `targetVal + 1` additions in `putTileAt`.
   - Removed `+ 1` array population transforms inside the `data2D` push function in `renderMap()`.
2. Initialized `currentTileType` globally and internally to `1` instead of `2` to respect that the absolute first tile of the tileset is index `1`.
3. Adjusted empty mapping and initializations across both `RpgMapEditor.tsx` and `RpgMode.tsx` to handle zeros properly (`Array(...).fill(0)` and `isEraser ? 0 : ...`).
4. Incorporated a UI refresh `RefreshCw` icon into the Tileset Left Sidebar (in `RpgMapEditor.tsx`) to trigger the pre-existing `loadTilesets` method seamlessly.
